# 投标 Markdown 输入规范

第一阶段要求投标技术文件先转换为 Markdown，再交给系统处理。系统会进行二次规范化，但输入越稳定，切章和来源召回越可靠。

## 文件要求

- 编码：UTF-8。
- 格式：`.md` 或 `.markdown`。
- 内容：以投标技术文件正文为主，尽量包含工程概况、火区勘查、治理工艺、质量安全、进度、成果评价等章节。
- 暂不支持：直接上传 PDF、Word、扫描件、图纸或压缩包。

## 标题规范

推荐使用 ATX 标题：

```markdown
# 宁夏贺兰山汝箕沟太西煤火区安全与生态治理项目一期工程北一火区治理施工项目技术文件

## 工程概况

### 项目地理位置与交通条件

### 自然地理情况

## 注水工程施工

## 钻孔与灌、注浆工程施工

## 覆盖封堵工程
```

也支持常见编号标题：

```markdown
1 工程概况
1.1 项目地理位置与交通条件
3.2.1 注水工程施工
3.2.2 钻孔与灌、注浆工程施工
```

标题建议：

- 每个标题单独占一行。
- 不要把正文和标题写在同一行。
- 标题层级应稳定递进，避免同一小节前后出现多个不同编号。

## 系统规范化行为

`MarkdownCanonicalizer` 会自动处理：

- 去除 UTF-8 BOM。
- 统一换行符。
- 去除 frontmatter。
- 统一 `#标题` 为 `# 标题`。
- 删除明显目录噪声，如 `1.1 火区位置........ 3`。
- 压缩连续空行。
- 保留正文表格、列表和关键参数。

## 切章规则

`MarkdownSectionSplitter` 会识别：

- `#` 到 `######` 标题。
- `第一章`、`第一节` 等中文章节标题。
- `1`、`1.1`、`3.2.1` 等编号标题。

每个切出的章节包含：

- `id`：稳定章节 ID。
- `title_path`：从父标题到当前标题的路径。
- `level`：标题层级。
- `content`：本节正文。
- `keywords`：火区、勘查、注水、钻孔、灌浆、覆盖封堵、质量、安全等关键词。
- `source_file`、`start_line`、`end_line`：来源定位信息。

## 推荐保留的火区治理核心内容

为了让施组模板小章节能自动召回来源，投标 Markdown 中建议保留以下章节或等价内容：

- 工程概况。
- 火区位置及交通条件。
- 自然地理、水文地质、工程地质。
- 火区勘查成果、温度、裂隙、塌陷、烟气、异常区。
- 治理范围、块段划分、灭火工程量、进度安排。
- 注水工程施工。
- 钻孔与灌、注浆工程施工。
- 覆盖封堵工程。
- 质量保证、安全管理、环境保护、文明施工。
- 灭火效果评价、监测、验收成果。
